
## Plugins

- everything-claudecode
- superpowers

1.使用/brainstorming命令把大致需求告诉它，让它调研然后互动明确细节，然后它会输出设计文档和实施文档，这两个文档它写完的时候你最好都review一下，有要改的立即让它改。写实施文档之前最好明确告诉它要用TDD方式，不然它有可能会忘。  
2.实施文档写完后它会问你采用subagent并行还是当前会话串行的方式进行实施，我一般都选1并行，因为方便，不用一次次确认，效果也还行。当然在实施的过程中根据不同的实施计划可能还是会中断，需要你响应或者互动，看情况。  
3.实施完成后，一般它自己单元测试也会全通过，你自己需要手动进行功能测试，然后发现问题让它再修改，直到功能全部通过。  
4.功能通过后，另开一个会话，使用/requesting-code-review命令对所有本次修改（git未提交的变更）进行代码审查，会给出一个报告，但是有时候可能会报出夸大其词的问题，所以再调用/receiving-code-review命令让它对上面的审查报告进行“怀疑性确认”（同一会话或者新会话都行，切换不同的模型也可以），二次确认后能保证审查出的问题基本上都是真的。  
5.对审查出的问题自己进行判断，需要修复哪些问题就让它一次性或者逐个修复就好。  
6.修复完之后，再进行步骤4-6的循环，直到你判断彻底没问题了。  
7.让它帮你提交代码，自动生成提交信息

防止 SuperPower 过分调用，在 AGENTS.md 中添加

```markdown
## Superpowers 本地覆写
- 轻量任务不进入完整 brainstorming / writing-plans / using-git-worktrees / subagentdriven-development 链路。
- 轻量任务定义：单文件或小范围修改、明确 bug 修复、配置调整、文案修改、小测试补充。
- 轻量任务默认直接分析代码并实现；只有遇到关键不确定性时才提问，且首次最多问 1 个问题。
- 如果项目上下文、AGENTS.md、现有代码已经能回答的问题，不要重复提问。
- 非我明确要求时，不要默认创建 worktree。
- 非我明确要求时，不要默认把 spec / plan 提交到 git。
- 在 Codex 环境中，默认优先使用 executing-plans，而不是 subagent-driven-development。
- 只有在任务明确适合并行、且平台对子代理支持良好时，才使用 subagent-driven-development。
- 需要确认时，优先一次性给出 2 到 3 个可选方案和推荐，不要把确认拆成过多轮。
- 以下操作仍然必须确认：删除文件、大规模重构、修改 git 历史、推送远程、改环境配置、改 CI、数据
库变更。
```
