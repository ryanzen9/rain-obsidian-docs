## 一、什么是 API

**API（Application Programming Interface）** 就是一个接口，通过这个接口可以帮你实现某种功能，让你不用亲自做辛苦又重复的工作。

> 就像天气热的时候，遥控一按冷气就有了，完全不用我们制作冰块来降温——**遥控就是这里的 API**。

计算机中的 API 本质上就是一些代码，REST API 也是按照特定架构风格编写的代码。

---

## 二、REST API 详解

### 2.1 REST 是什么

REST 是一种**架构风格**，不是协议，也不是语言。

> 打个比方：就像穿衣服，并没有法律规定格子衫加牛仔裤一定是程序员，但因为这样的穿搭形成了容易辨认的风格，大家看到第一眼就会想到是程序员风格。

REST 架构风格也一样，有规律可循。

### 2.2 REST 的工作流程

假设用户访问了一个天气预报的网页应用：

1. 天气状态保存在数据库里
2. 用户想看某月某日的天气
3. 网页里含有调用 API 的代码，向 API 发送请求
4. API 根据请求在数据库里拿资源，并返回给客户端

### 2.3 REST 的核心含义

**Representational（表现层）**：
- 资源在返回给客户端时，可以有多种表现形式
- 常见的有 **JSON** 和 **XML**
- 让资源有某种表现形式，就是 Representational 的含义

**State Transfer（状态传输）**：
- REST 架构要求**客户端的状态不保存在服务端**
- 每一次请求都需要正确表示自己的状态，并传输给服务端
- 把客户端当前的状态通过 API 请求传递给服务端

### 2.4 REST 的六个约束

| 约束 | 说明 |
|------|------|
| **客户端-服务端分离** | 客户端和服务端各自独立，客户端不需要知道服务端的具体逻辑，服务端不需要知道客户端的用户界面 |
| **统一接口** | 不同的资源用不同的 URI 表示；资源通过 XML 或 JSON 等方式管理；请求时需要表明要执行的操作 |
| **无状态** | 服务端不保存客户端状态，提高了服务端性能 |
| **可缓存** | 客户端可以缓存响应，减少网络带宽和响应时间 |
| **分层系统** | 可以经过负载均衡、多个服务器形成分层结构，增强安全性和可扩展性 |
| **按需代码**（可选） | 服务端可以响应可执行代码（如 JavaScript）给客户端，扩展客户端功能 |

### 2.5 REST 的缺点

1. **过度依赖 API 文档**：调用 REST API 就像盲人摸象，不知道接口细节很难使用
2. **过度获取或获取不足**：一个接口返回太多数据（浪费带宽）或太少数据（需要多次请求）
3. **端点管理复杂**：客户端需要面对多个资源端点，端点变了也要做相应更新

## 三、GraphQL 详解

### 3.1 GraphQL 解决了什么问题

GraphQL 的出现解决了 REST 的很多问题：

1. **自文档化**：提供图形界面进行请求，几乎不需要 API 文档，大大减少前后端沟通成本
2. **Schema 查询**：有 schema 可供查询，数据类型一目了然
3. **精确获取数据**：可以精确指定需要的数据字段，避免过度获取或获取不足

### 3.2 GraphQL 的核心特点

- **单一端点**：客户端只需要面对一个端点，而不是多个资源端点
- **灵活查询**：可以把所有需要整合的资源放在一起查询，类似数据库查询语法
- **嵌套支持**：支持嵌套查询，不需要客户端另外进行数据关联

### 3.3 GraphQL 的三种操作

| 操作 | 说明 | 对应功能 |
|------|------|----------|
| **Query** | 查询 | 对应 REST 的 GET，用于读取数据 |
| **Mutation** | 变更 | 对应 REST 的 POST/PUT/DELETE，用于写入数据 |
| **Subscription** | 订阅 | 客户端和服务端建立持久连接，事件触发时推送数据，利于实时应用 |

> 注意：这三种操作只是语义上的，实际上可以创建删除数据的 query，就像 HTTP 的方法和状态码一样，实际看到的和语义上可能会有出入。

### 3.4 GraphQL 查询语法示例

```graphql
query GetUser($id: ID!) {
  user(id: $id) {
    name
    email
    posts {
      title
      content
    }
  }
}
```

结构说明：
1. **操作类型**：query 或 mutation
2. **操作名**（可选）：便于调试和日志记录
3. **参数**：传入可复用的变量
4. **返回字段**：大括号内指定服务器要响应的内容，支持嵌套

### 3.5 什么时候选择 GraphQL

**REST 更适合**：
- 简单的 API 和应用
- 数据结构不复杂，多几次请求也无所谓
- 初创企业，开发成本相对不高

**GraphQL 更适合**：
- 拥有复杂关系的数据
- 有强类型的需求
- 面对很多移动端用户，对客户端性能有较高要求
- 可以承受一定的学习成本

> 可以从 REST 逐渐过渡到 GraphQL，也可以两者都用。

## 四、gRPC 详解

### 4.1 RPC 技术机制

**RPC（Remote Procedure Call）** 是一种技术机制：

- 将应用的不同服务拆分开来，部署在不同的服务器上
- 各自把服务封装起来，只提供函数接口
- 调用某个服务时，**就像调用本地代码一样**
- 网络通信的复杂事情都不用管

### 4.2 RPC 的三个核心要素

1. **Functions**：服务端有相应的函数，提供某些功能
2. **Stub**：客户端调用服务端的代理函数
3. **Marshal**：编码/解码过程，相当于翻译

### 4.3 gRPC 与 Protocol Buffers

实现 RPC 可以使用框架，比如 **gRPC**。gRPC 需要遵守某种协议，即 **Protocol Buffers**：

- 用 `.proto` 后缀的文件进行定义
- 定义 **service** 时，就是定义服务名称和 RPC 端点
- 定义 **message** 时，就是定义数据请求和响应

### 4.4 gRPC 的优势

| 优势 | 说明 |
|------|------|
| **高效传输** | Protocol Buffers 序列化成二进制传输，比 JSON 负载更小，解析消耗更少资源 |
| **流式传输** | 支持单向和双向流式传输，基于 HTTP/2 构建 |
| **HPACK 压缩** | HTTP/2 的 HPACK 压缩进一步降低负载体积 |

### 4.5 gRPC 的局限性

- **浏览器支持**：目前还不能直接从浏览器调用 gRPC 服务（因为基于 HTTP/2）
- **语言支持**：目前还不支持全部编程语言
- **适用场景**：常见于分布式系统的微服务中，即**服务器与服务器之间的交流**

## 五、tRPC 详解

### 5.1 什么是 tRPC

如果前后端都用 **TypeScript**，**tRPC** 是一个很好的选择。

> 注意：tRPC 和 gRPC **不是同一个东西**，tRPC 没有像 Protocol Buffers 这样的定义文件。

### 5.2 tRPC 的优势

| 优势 | 说明 |
|------|------|
| **类型安全** | 客户端代码中有服务端不存在的字段，通过编辑器就可以马上看出问题 |
| **开发体验** | 前后端合在一起的全栈开发者，tRPC 可以降低很多开发难度 |
| **库的统一** | 前后端只需要专注于 tRPC 这组库来进行开发 |

### 5.3 与 GraphQL 的对比

GraphQL 会有很多库（如 Apollo Client、Relay 等），每个库的优缺点不一样，选择太多有时候也未必是好事。

tRPC 提供了一组统一的库，对于全栈开发者更加友好。

## 六、技术选型总结

| 技术 | 适用场景 | 特点 |
|------|----------|------|
| **REST** | 简单 API、初创企业、快速开发 | 简单直观，文档成熟，但容易过度/不足获取 |
| **GraphQL** | 复杂数据关系、移动端应用、强类型需求 | 精确获取数据，自文档化，支持实时订阅 |
| **gRPC** | 微服务架构、服务器间通信、高性能需求 | 二进制传输高效，支持流式，但不支持浏览器直接调用 |
| **tRPC** | 全栈 TypeScript 项目 | 端到端类型安全，开发体验好，适合前后端同构项目 |

---

## 七、结语

选择 API 技术时，需要考虑：

1. **项目复杂度**：简单项目 REST 足矣，复杂项目考虑 GraphQL 或 gRPC
2. **团队结构**：前后端分离团队适合 REST/GraphQL，全栈团队适合 tRPC
3. **性能要求**：移动端或高并发场景考虑 GraphQL 或 gRPC
4. **技术栈**：TypeScript 全栈可以优先考虑 tRPC

没有最好的技术，只有最适合的技术。
