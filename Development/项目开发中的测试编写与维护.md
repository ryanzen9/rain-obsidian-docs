# 项目开发中的测试编写与维护

软件测试是保障代码质量的核心手段。本文从测试金字塔出发，依次梳理单元测试、集成测试、端对端测试的适用范围与编写方法，并补充回归测试、性能测试以及测试驱动开发（TDD）的实践要点。

## 金字塔（Test Pyramid）原则

测试金字塔由 Mike Cohn 提出，是指导测试策略分层的核心模型。它按粒度与运行成本将测试分为三层：

- **底层（大量）：单元测试**——快速、稳定、运行成本低，覆盖函数与方法的逻辑正确性。
- **中间（适量）：集成测试**——验证模块间、服务间、与外部依赖的交互是否按契约工作。
- **顶层（少量）：端到端测试（E2E）**——模拟真实用户场景，成本高、速度慢、易受环境影响，但最接近业务真相。

越靠近底层，越关注**代码正确性**；越往顶层，越关注**业务/用户行为正确性**。一个健康的测试策略应当让大部分测试落在底层、少量落在顶层，避免形成"倒金字塔"（E2E 过多、单元测试缺失）或"冰淇淋筒"（大量手工测试、少量自动化）形状的分布。

## 单元测试 (Unit Testing)

单元测试用于验证单个函数或方法在隔离环境下的行为是否符合预期。它的核心思想是：将被测代码与外部依赖（数据库、网络、文件系统）隔离开，只关注该代码块自身的逻辑。一个函数或方法可以对应一组单元测试，称为测试用例；一整套测试用例覆盖该代码块的全部预期行为。

编写单元测试时，通常围绕三个维度展开：

- **逻辑检查**：验证正常输入下输出是否匹配业务规则。
- **边界处理**：覆盖零值、空值、极大/极小值、未知标识符等边缘情况。
- **错误处理**：确认非法输入下代码是否以预期方式失败（抛出指定异常或返回错误码）。

一个广为采用的测试结构是 **AAA 模式**——Arrange（准备数据与依赖）、Act（执行被测行为）、Assert（断言结果）。下面以 TypeScript + Jest 为例，展示一个折扣计算函数及其完整测试。

被测函数：

```typescript
// discount.ts
export function calculateDiscount(
  amount: number,
  discountCode?: string
): number {
  if (amount < 0) {
    throw new Error('订单金额不能为负数');
  }

  if (discountCode === 'SAVE10') return amount * 0.9;
  if (discountCode === 'SAVE50') return amount * 0.5;

  return amount;
}
```

测试用例覆盖逻辑验证、边界处理和错误处理：

```typescript
// discount.test.ts
import { calculateDiscount } from './discount';

describe('calculateDiscount', () => {
  // 逻辑检查
  it('应用 SAVE10 折扣码，打九折', () => {
    expect(calculateDiscount(100, 'SAVE10')).toBe(90);
  });

  it('应用 SAVE50 折扣码，打五折', () => {
    expect(calculateDiscount(200, 'SAVE50')).toBe(100);
  });

  it('不传折扣码时返回原价', () => {
    expect(calculateDiscount(100)).toBe(100);
  });

  // 边界处理
  it('金额为 0 时计算结果为 0', () => {
    expect(calculateDiscount(0, 'SAVE10')).toBe(0);
  });

  it('传入未知折扣码时返回原价', () => {
    expect(calculateDiscount(100, 'XYZ')).toBe(100);
  });

  // 错误处理
  it('金额为负数时抛出异常', () => {
    expect(() => calculateDiscount(-50)).toThrow('订单金额不能为负数');
  });
});
```

常用的单元测试工具包括 Jest、Vitest、Mocha 等（JavaScript/TypeScript 生态）以及 JUnit、pytest、Go testing 等（其他语言生态）。选择工具时优先考虑与项目技术栈的集成便利性以及运行速度。

## 集成测试 (Integration Testing)

单元测试确保每个模块自身正确，但模块组合在一起时可能出现契约不匹配、序列化错误或事务回滚问题。集成测试的目标就是验证模块间的协作是否正常。

集成测试的特点是"部分真实"——不 mock 内部模块，让它们真实交互；只对外部边界（如第三方 API、消息队列、支付网关）进行模拟或替换。这样做比单元测试重，但比端到端测试轻，能捕捉到单元测试无法发现的接口契约与数据一致性问题。

下面的例子展示 UserService 与 UserRepository 之间的集成测试。用内存实现替代真实数据库，在保持测试快速且可重复的前提下验证二者的协作逻辑。

```typescript
// user.repository.ts
export interface User {
  id: number;
  email: string;
  name: string;
}

export interface UserRepository {
  findByEmail(email: string): Promise<User | null>;
  save(user: Omit<User, 'id'>): Promise<User>;
}
```

```typescript
// user.service.ts
import { UserRepository, User } from './user.repository';

export class UserService {
  constructor(private readonly repo: UserRepository) {}

  async register(email: string, name: string): Promise<User> {
    const existing = await this.repo.findByEmail(email);
    if (existing) {
      throw new Error('该邮箱已被注册');
    }
    return this.repo.save({ email, name });
  }
}
```

```typescript
// user.service.test.ts
import { UserService } from './user.service';
import { UserRepository, User } from './user.repository';

class InMemoryUserRepo implements UserRepository {
  private users: User[] = [];
  private nextId = 1;

  async findByEmail(email: string): Promise<User | null> {
    return this.users.find((u) => u.email === email) ?? null;
  }

  async save(user: Omit<User, 'id'>): Promise<User> {
    const created = { ...user, id: this.nextId++ };
    this.users.push(created);
    return created;
  }
}

describe('UserService 集成测试', () => {
  let service: UserService;
  let repo: InMemoryUserRepo;

  beforeEach(() => {
    repo = new InMemoryUserRepo();
    service = new UserService(repo);
  });

  it('成功注册新用户', async () => {
    const user = await service.register('alice@example.com', 'Alice');
    expect(user.id).toBeDefined();
    expect(user.email).toBe('alice@example.com');
  });

  it('重复邮箱注册时抛出异常', async () => {
    await service.register('bob@example.com', 'Bob');
    await expect(
      service.register('bob@example.com', 'Bobby'),
    ).rejects.toThrow('该邮箱已被注册');
  });
});
```

在实际项目中，若使用 ORM（如 TypeORM、Prisma），集成测试通常会连接一个测试专用数据库（如 SQLite 内存模式或 Docker 中的临时 PostgreSQL）。关键在于确保测试可重复、彼此隔离，且不依赖外部环境的状态。

## 端对端测试（E2E Testing）

端对端测试覆盖从用户入口到最终响应的完整业务流程，例如"打开登录页 → 输入凭据 → 提交表单 → 跳转到首页"这一完整路径。它最接近真实用户的操作方式，但运行成本也最高：速度慢、依赖完整环境、容易因网络或第三方服务波动而失败。

因此端对端测试只应聚焦于核心用户旅程（登录、注册、下单、支付），不应试图覆盖所有分支逻辑——分支逻辑交给单元测试和集成测试处理。下面是一个使用 Playwright 编写的登录流程测试。

```typescript
// e2e/login.spec.ts
import { test, expect } from '@playwright/test';

test.describe('登录流程', () => {
  test('使用正确凭据登录成功', async ({ page }) => {
    await page.goto('/login');
    await page.fill('[data-testid="email"]', 'user@example.com');
    await page.fill('[data-testid="password"]', 'correct-password');
    await page.click('[data-testid="submit"]');

    await expect(page).toHaveURL('/dashboard');
    await expect(page.locator('[data-testid="greeting"]')).toContainText('欢迎回来');
  });

  test('错误凭据时停留在登录页并显示提示', async ({ page }) => {
    await page.goto('/login');
    await page.fill('[data-testid="email"]', 'wrong@example.com');
    await page.fill('[data-testid="password"]', 'wrong');
    await page.click('[data-testid="submit"]');

    await expect(page).toHaveURL('/login');
    await expect(page.locator('[data-testid="error"]')).toBeVisible();
  });
});
```

推荐使用 `data-testid` 属性作为选择器而非 CSS class 或 DOM 结构——UI 样式与结构的变动不会破坏测试。主流的端对端测试工具包括 Playwright、Cypress 和 Selenium，其中 Playwright 因其多浏览器支持和自动等待机制而受到广泛采用。

## 回归测试 (Regression Testing)

回归测试的目标是确保新代码不会破坏已有功能。它不是特定的测试类型，而是一种测试策略：每当代码发生变更（新功能、Bug 修复、重构），重新运行已有的测试套件，验证既有行为未被改变。

在实践中，回归测试通常由 CI/CD 流水线在每次提交或合并请求时自动执行。任何测试类型（单元测试、集成测试、端到端测试）都可以充当回归测试，只要它们在代码变更后被重复运行。典型的流程是：开发者提交代码 → CI 触发 → 运行完整测试套件 → 有失败则阻塞合并。这让团队能在引入 Bug 的第一时间发现并修复，而不是等到上线后由用户报告。

快照测试（Snapshot Testing）是回归测试在前端开发中的一种常见形式——它将组件的渲染输出保存为快照文件，后续运行时对比当前输出与快照的差异，意外差异即视为回归。

```typescript
// 快照测试示例：React 组件
import { render } from '@testing-library/react';
import { UserProfile } from './UserProfile';

it('UserProfile 渲染结果与快照一致', () => {
  const { container } = render(
    <UserProfile name="Alice" email="alice@example.com" />
  );
  expect(container.firstChild).toMatchSnapshot();
});
```

注意，快照测试不能替代行为断言——它只告诉你"变了"，不告诉你"变得对不对"。快照内容应小而聚焦，过大的快照容易在 review 中被忽略，沦为形式。

## 性能测试 (Performance Testing)

性能测试衡量系统在特定负载下的响应时间、吞吐量和资源消耗，而非功能的对错。它通常在发布前、架构变更后或大促活动前按需执行，而不在每次提交时运行（成本过高）。结果应与历史基线对比，以判断是否存在性能退化。

常见的性能测试形态包括：

- **负载测试（Load Testing）**：模拟正常峰值并发用户量，验证系统在预期负载下能否保持可接受的响应时间。
- **压力测试（Stress Testing）**：持续增加负载直到系统崩溃，找到系统的极限容量和瓶颈点。
- **耐久测试（Endurance Testing）**：在中等负载下长时间运行，观察是否存在内存泄漏、连接池耗尽等渐进性问题。

在 Node.js 生态中，常用的性能测试工具有 autocannon（HTTP 基准测试）、Artillery 和 k6。下面是一个使用 k6 的简单示例：

```javascript
// load-test.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 50,          // 模拟 50 个并发用户
  duration: '30s',  // 持续 30 秒
};

export default function () {
  const res = http.get('https://api.example.com/users');
  check(res, {
    '状态码为 200': (r) => r.status === 200,
    '响应时间 < 500ms': (r) => r.timings.duration < 500,
  });
  sleep(1);
}
```

## 测试驱动开发（TDD）

TDD（Test-Driven Development）将测试的顺序颠倒过来：**先写测试，再写代码**。这不只是测试技术的改变，更是一套设计方法论——测试驱动你思考接口的契约、依赖的边界，以及"什么才算完成"。

| 传统开发流程 | TDD 流程 |
|---|---|
| 思考功能 → 写代码 → 运行看效果 → 补测试（也许） → 发现 Bug 再改 | 把需求转化为测试 → 运行测试（失败） → 写最少代码通过 → 重构优化 |

TDD 的核心循环称为 **Red-Green-Refactor**：先写一个恰好失败的测试（Red），再写刚好让测试通过的最少代码（Green），最后在测试全部通过的前提下优化代码结构（Refactor）。下面以一个 ShoppingCart 类为例，展示三轮迭代。

**第 1 轮：空购物车。** 先写下第一个测试——新创建的购物车商品数量应为 0。运行，测试失败（Red），因为 ShoppingCart 类还不存在。创建类并让 `count` 返回 0，测试通过（Green）。

```typescript
// cart.test.ts（第 1 轮新增）
it('新购物车商品数为 0', () => {
  const cart = new ShoppingCart();
  expect(cart.count).toBe(0);
});

// cart.ts（第 1 轮实现）
export class ShoppingCart {
  get count(): number {
    return 0;
  }
}
```

此时 `return 0` 是明显的"假实现"，但 TDD 允许这样做——它让下一个测试来推动真正的逻辑。

**第 2 轮：添加商品。** 新增测试：添加一件商品后，数量应变为 1。第二个测试失败（Red），因为 `add` 方法不存在。实现 `add` 方法和真实的 `count`，两个测试同时通过（Green）。

```typescript
// cart.test.ts（第 2 轮新增）
it('添加商品后数量增加', () => {
  const cart = new ShoppingCart();
  cart.add({ name: 'TypeScript 实战', price: 59 });
  expect(cart.count).toBe(1);
});

// cart.ts（第 2 轮实现）
export class ShoppingCart {
  private items: Array<{ name: string; price: number }> = [];

  get count(): number {
    return this.items.length;
  }

  add(item: { name: string; price: number }): void {
    this.items.push(item);
  }
}
```

**第 3 轮：计算总价。** 新增测试：添加两件商品后，总价应为两者之和。运行失败（Red），实现 `total` 属性使测试通过（Green）。最后，提取 `CartItem` 接口消除重复的类型字面量——这是 Refactor 步骤，测试保持全绿。

```typescript
// cart.test.ts（第 3 轮新增）
it('计算购物车总价', () => {
  const cart = new ShoppingCart();
  cart.add({ name: 'A', price: 30 });
  cart.add({ name: 'B', price: 70 });
  expect(cart.total).toBe(100);
});

// cart.ts（第 3 轮实现 + 重构）
interface CartItem {
  name: string;
  price: number;
}

export class ShoppingCart {
  private items: CartItem[] = [];

  get count(): number {
    return this.items.length;
  }

  get total(): number {
    return this.items.reduce((sum, item) => sum + item.price, 0);
  }

  add(item: CartItem): void {
    this.items.push(item);
  }
}
```

经过三轮迭代，我们得到了一个有测试保障的 ShoppingCart 类。过程中每一步的代码变更都有测试作为安全网，重构时不必担心破坏已有功能。

TDD 的采用需要权衡。它的前期投入较大，因为测试代码的体量通常超过业务代码；但当项目规模扩大后，充足的测试用例使得新功能开发与重构没有后顾之忧。对于需求尚不稳定的早期项目，可以优先在核心业务逻辑上应用 TDD，而在 CRUD 或简单转发层保持实用主义的测试策略。TDD 的目标是保证功能与设计，并不追求 100% 的测试覆盖率——强行覆盖 getter/setter 或纯转发函数没有意义，只驱动有逻辑、有行为的代码。此外，TDD 天然产生"可测试"的设计：那些难以测试的代码（紧耦合、静态调用）你在写之前就会避开，代码自然向 SOLID 原则靠拢，尤其是依赖倒置和单一职责。
