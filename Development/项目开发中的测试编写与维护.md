## 金字塔（Test Pyramid）原则

- **底层（大量）：单元测试**  
    快速、稳定、运行成本低。
    
- **中间（适量）：集成测试 / 服务测试 / API测试**  
    验证模块间、服务间、与外部依赖的交互。
    
- **顶层（少量）：端到端测试（E2E） / UI测试**  
    模拟真实用户场景，成本高、速度慢、容易受环境影响。
    
越靠近底层越注重“**代码正确性**”，越往上越注重“**业务/用户行为正确性**”。
## 单元测试 (Unit Testing)

- 测试范围
	- 单元测试_是一个代码块，用于验证较小、孤立的应用程序代码块（通常是函数或方法）的准确性。 to be 检查方法或者类的结果是否符合预期。
	- 单个代码块也可以有一组单元测试，称为_测试用例_。一整套测试用例涵盖了代码块的全部预期行为，但并不总是需要定义完整的测试用例集。
- 关注点
	- 逻辑检查，检查执行结果是否符合业务预期
	- 边界处理，对于给定的输入，系统如何响应？ 系统如何响应典型输入、边缘用例或无效输入？
	- 错误处理，当输入中出现错误时，系统会如何响应？
- 编写原则
	- 当一个代码块的运行需要系统的其他部分时，不能将单元测试与此外部数据结合使用。单元测试需要隔离运行。（集成测试）
- 工具例： Jest， Vitest， Junit
- 用例模拟
	- Arrange
	- Act
	- Assert
- 单元测试编写示范
## 集成测试 (Integration Testing)

- 测试范围： 多个模块间的协作。例如：一个 Service 类调用了 Repository 类，验证它们组合在一起是否能正确处理数据。 
- 关注点： 关注**契约**。模拟（Mock）掉最外层的外部服务（如 Stripe API），但内部逻辑保持真实。接口契约是否正确，序列化/反序列化，事务回滚，数据一致性问题。

## 端对端测试（E2E Testing）

- **范围**：整个系统从用户入口到最终响应的完整业务流程。如前端页面操作→后端服务→数据库→外部服务。
- 用户核心旅程能否跑通（例如登录、下单、支付全流程）。
- 关注点： 更接近于最终用户与生产系统交互的方式，模拟应用程序启动到请求由接口到执行完成
## 测试驱动的开发（TDD）

**TDD（测试驱动开发）**，意思是“先写测试，再写代码”。你可能觉得这有点反直觉——不是应该先把功能写完，再测试对不对吗？

TDD 恰恰把这个顺序颠倒了过来。它不只是测试技术，更是一套**设计与开发的方法**。

所谓TDD，是指先编写接口，紧接着编写测试。编写完测试后，我们才开始真正编写实现代码。在编写实现代码的过程中，一边写，一边测，什么时候测试全部通过了，那就表示编写的实现完成。

|传统开发流程|TDD 流程|
|---|---|
|思考功能 → 写代码 → 运行看效果 → 补测试（也许） → 发现Bug再改|把需求转化为测试 → 运行测试（失败） → 写最少代码让它通过 → 重构优化|
原则：
	- 先编写测试用例，再编写业务代码进行实现。
	- 只写刚好让测试失败的单元测试
	- 在能保证测试通过前提下，不多编写任何一行多余代码。
	- 编写用例 -> 编写业务代码 -> 通过用例 -> 编写用例 进行逐步循环开发。

优点：
- 天然有充足的测试用例，对于 Feature 的开发 与 重构没有负担。
