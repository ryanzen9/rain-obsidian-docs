## 为什么需要加密

在数字时代，数据安全至关重要。无论是用户的登录密码、支付信息，还是企业的商业机密，一旦被未授权访问，后果可能是隐私泄露、财产损失甚至法律追责。加密是保护数据安全的核心技术——它将可读的明文转换为无法理解的密文，只有持有正确密钥的人才能还原。

**重要提示：永远不要尝试自行设计加密算法。** 密码学是一个高度专业化的领域，设计安全的算法需要深厚的数学基础和密码学知识。业余设计的算法往往存在严重漏洞，容易被破解。我们应该始终选择经过密码学界广泛审查和认可的标准算法与实现库。

本文将重点介绍如何使用 **PBKDF2** 进行密钥派生，以及使用 **AES-GCM** 进行数据加密和解密，并结合 Node.js 的 `crypto` 模块给出完整的代码示例。

## 核心概念

在动手之前，先理解几个关键概念，它们贯穿整个加密流程：

**对称加密（Symmetric Encryption）**：加密和解密使用同一把密钥。AES 是目前最广泛使用的对称加密算法。它的优势是速度快、开销小，适合对大量数据进行加密。挑战在于如何安全地分发和保管这把密钥。

**密钥派生函数（Key Derivation Function, KDF）**：用户输入的密码往往是弱密码（如 `123456`），直接用作加密密钥非常不安全。KDF 的作用是将弱密码转化为固定长度、足够随机的强密钥。PBKDF2 是最常用的 KDF 之一。

**盐（Salt）**：一段随机生成的数据，与密码一起输入 KDF。盐的作用是防止彩虹表攻击——即使两个用户使用了相同的密码，不同的盐也会产生完全不同的密钥。盐不是秘密，可以与密文一起明文存储。

**初始化向量（Initialization Vector, IV）**：一段随机生成的数据，确保即使使用相同密钥加密相同明文，每次产生的密文也不同。IV 也不是秘密，需要与密文一起存储，解密时需要用到。

**认证标签（Authentication Tag, AuthTag）**：由认证加密算法（如 AES-GCM）生成的一段数据，用于验证密文没有被篡改。AuthTag 必须与密文一起存储，解密时用于完整性校验。

**AES-GCM（Advanced Encryption Standard - Galois/Counter Mode）**：AES 的一种运行模式。GCM 模式同时提供加密和认证两种能力——它不仅保护数据的保密性（加密），还保护数据的完整性（认证）。任何对密文的篡改都会在解密时被检测出来。

**PBKDF2（Password-Based Key Derivation Function 2）**：一种标准的密钥派生函数，通过反复迭代哈希运算（如 SHA-256），将弱密码转化为强密钥。迭代次数越高，攻击者暴力破解的代价越大。

## 三步保障数据安全

数据安全保障可以分解为三个步骤：先用 PBKDF2 从用户密码派生强密钥，再用 AES-GCM 加密数据，最后用 AES-GCM 解密并验证数据。

### 步骤 1：密钥派生——使用 PBKDF2

用户密码（如 PIN 码）通常太短、太简单，直接用于加密极不安全。PBKDF2 的作用就像一个"密码强化器"：它将用户密码与随机盐拼接后，反复进行哈希运算（如 10 万次 SHA-256），最终输出一个 32 字节的强密钥，可以直接用于 AES-256 加密。

盐的加入确保了即使两个用户使用相同的密码，派生出的密钥也完全不同，攻击者无法使用预计算的彩虹表来破解。迭代次数则增加了暴力破解的计算代价——攻击者每尝试一个密码，都需要付出 10 万次哈希运算的成本。

```javascript
const crypto = require("crypto");

// 从密码派生加密密钥
function deriveKeyFromPassword(password, salt) {
  const iterations = 100000; // 迭代次数，至少 10 万次
  const keyLength = 32; // AES-256 需要 32 字节密钥
  const digest = "sha256"; // 哈希算法

  return new Promise((resolve, reject) => {
    crypto.pbkdf2(password, salt, iterations, keyLength, digest, (err, derivedKey) => {
      if (err) reject(err);
      else resolve(derivedKey);
    });
  });
}

// 生成随机盐（每个用户或每份数据应使用不同的盐）
const salt = crypto.randomBytes(16);

async function main() {
  const userPassword = "123456"; // 用户的弱密码
  const derivedKey = await deriveKeyFromPassword(userPassword, salt);
  console.log("派生密钥:", derivedKey.toString("hex"));
  // 后续使用 derivedKey 进行加密操作
}

main();
```

几个关键参数的说明：

- `iterations`：哈希运算的重复次数。数值越高越安全，但密钥派生过程也越慢。建议至少 10 万次，生产环境可根据硬件性能调高到 30 万次以上。
- `keyLength`：生成的密钥长度。32 字节对应 AES-256，16 字节对应 AES-128。
- `digest`：内部使用的哈希算法，SHA-256 是安全且广泛使用的选择。
- `crypto.randomBytes(16)`：生成 16 字节的随机盐。盐必须是随机且唯一的，每个用户或每份数据使用不同的盐。

### 步骤 2：加密数据——使用 AES-GCM

有了派生密钥之后，就可以用它来加密数据了。AES-GCM 是目前推荐的对称加密模式，它在加密的同时生成认证标签（AuthTag），确保密文不会被篡改。

每次加密都必须生成一个新的随机 IV。这保证了即使使用相同密钥加密相同明文，每次产生的密文也完全不同——攻击者无法通过对比密文来推断明文之间的关系。

```javascript
function encryptData(plaintext, derivedKey) {
  const iv = crypto.randomBytes(12); // GCM 模式推荐 12 字节 IV
  const cipher = crypto.createCipheriv("aes-256-gcm", derivedKey, iv);

  let encrypted = cipher.update(plaintext, "utf8", "base64");
  encrypted += cipher.final("base64");
  const authTag = cipher.getAuthTag(); // 获取认证标签

  return {
    iv: iv.toString("base64"),
    encryptedData: encrypted,
    authTag: authTag.toString("base64"),
  };
}
```

加密完成后，需要将 **盐（salt）、IV、密文（encryptedData）和认证标签（authTag）** 全部存储起来。其中盐、IV 和 AuthTag 不是秘密，可以明文存储；密文自然是加密后的数据。**绝对不要存储用户的原始密码或派生密钥。**

### 步骤 3：解密数据——使用 AES-GCM

解密是加密的逆过程。需要提供与加密时相同的派生密钥、IV 和 AuthTag。AES-GCM 解密器会先验证 AuthTag 是否匹配——如果密文被篡改，或者使用了错误的密钥/IV，验证就会失败并抛出错误。这种"先验证、再解密"的机制保证了数据的完整性和真实性。

```javascript
function decryptData(encryptedData, derivedKey, ivBase64, authTagBase64) {
  const iv = Buffer.from(ivBase64, "base64");
  const authTag = Buffer.from(authTagBase64, "base64");
  const decipher = crypto.createDecipheriv("aes-256-gcm", derivedKey, iv);

  decipher.setAuthTag(authTag); // 设置认证标签，用于完整性验证

  let decrypted = decipher.update(encryptedData, "base64", "utf8");
  decrypted += decipher.final("utf8"); // 验证失败会在此处抛出错误

  return decrypted;
}
```

如果 AuthTag 验证失败，`decipher.final()` 会抛出错误。这意味着数据可能被篡改，或者密钥不正确。在 catch 块中处理这种情况时，不要泄露具体的错误细节给客户端。

## 完整示例

将三个步骤串联起来，形成一个完整的加密-解密流程：

```javascript
const crypto = require("crypto");

// 密钥派生
function deriveKeyFromPassword(password, salt) {
  return new Promise((resolve, reject) => {
    crypto.pbkdf2(password, salt, 100000, 32, "sha256", (err, key) => {
      if (err) reject(err);
      else resolve(key);
    });
  });
}

// 加密
function encryptData(plaintext, derivedKey) {
  const iv = crypto.randomBytes(12);
  const cipher = crypto.createCipheriv("aes-256-gcm", derivedKey, iv);
  let encrypted = cipher.update(plaintext, "utf8", "base64");
  encrypted += cipher.final("base64");
  const authTag = cipher.getAuthTag();

  return {
    iv: iv.toString("base64"),
    encryptedData: encrypted,
    authTag: authTag.toString("base64"),
  };
}

// 解密
function decryptData(encryptedData, derivedKey, ivBase64, authTagBase64) {
  const iv = Buffer.from(ivBase64, "base64");
  const authTag = Buffer.from(authTagBase64, "base64");
  const decipher = crypto.createDecipheriv("aes-256-gcm", derivedKey, iv);
  decipher.setAuthTag(authTag);

  let decrypted = decipher.update(encryptedData, "base64", "utf8");
  decrypted += decipher.final("utf8");
  return decrypted;
}

// 端到端演示
async function main() {
  const password = "my-secret-pin";
  const salt = crypto.randomBytes(16);
  const plaintext = "这是一段需要加密的敏感数据";

  // 1. 密钥派生
  const derivedKey = await deriveKeyFromPassword(password, salt);
  console.log("派生密钥:", derivedKey.toString("hex"));

  // 2. 加密
  const { iv, encryptedData, authTag } = encryptData(plaintext, derivedKey);
  console.log("IV:", iv);
  console.log("密文:", encryptedData);
  console.log("认证标签:", authTag);

  // 存储：salt, iv, encryptedData, authTag（不存储密码和密钥）
  // ...

  // 3. 解密（从存储中读取 salt, iv, encryptedData, authTag）
  const derivedKey2 = await deriveKeyFromPassword(password, salt);
  const decrypted = decryptData(encryptedData, derivedKey2, iv, authTag);
  console.log("解密结果:", decrypted); // "这是一段需要加密的敏感数据"
}

main().catch(console.error);
```

注意：解密时需要用相同的密码和盐重新派生密钥。这意味着存储时必须保存盐，但不需要保存密钥——每次解密时从密码和盐重新派生即可。

## 安全注意事项

**密码强度仍然重要。** 虽然 PBKDF2 增加了暴力破解的代价，但弱密码仍然是整个链条中最薄弱的环节。尽可能引导用户选择更复杂的密码，或在应用层面强制密码复杂度要求。

**迭代次数要足够高。** PBKDF2 的安全性直接取决于迭代次数。建议至少 10 万次，生产环境可根据服务器性能调高。迭代次数过低会让攻击者可以快速暴力破解。

**盐和 IV 必须唯一且随机。** 每个用户或每份数据都应使用不同的盐，每次加密都应生成新的 IV。重复使用盐或 IV 会严重削弱安全性——相同密码加相同盐会产生相同的密钥，相同密钥加相同 IV 加相同明文会产生相同的密文。

**安全存储加密产物。** 盐、IV 和 AuthTag 不是秘密，可以与密文一起明文存储。但**绝对不要存储用户的原始密码或派生密钥**。解密时从密码和盐重新派生密钥即可。

**IV 长度选择。** AES-GCM 推荐使用 12 字节（96 位）的 IV。这个长度是 NIST 为 GCM 模式推荐的，更长的 IV 会被 GCM 内部额外哈希处理，更短的 IV 则会降低安全性。

**密钥生命周期管理。** 如果怀疑密钥泄露，应立即更换盐并重新加密所有数据。考虑实现密钥轮换机制，定期更新密钥。
