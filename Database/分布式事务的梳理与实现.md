
## 分布式事务是什么？

通常代码中的事务是由内存总线与数据库的连接控制的。当微服务，或者与第三方服务交互的场合，则需要使用分布式事务对一致性进行保证。

**分布式事务：** **在分布式系统中，一个业务操作需要跨多个服务、数据库或节点协同完成，并且这些操作要么全部成功，要么全部失败回滚，从而保证整体数据一致性。**

### 事务的 ACID 特性

A： 原子性，事务是一个不可分割的最小执行单元，事务中的所有操作要么**全部成功**，要么**全部失败回滚**，不存在中间状态。

C： 一致性，事务执行前后，数据库必须从一个**合法状态**变换到另一个**合法状态**，所有约束（外键、唯一性、CHECK约束、触发器等）都必须满足。

I： 隔离性，多个事务并发执行时，一个事务的执行**不应被其他事务干扰**，每个事务感觉就像在独占数据库一样。

D： 持久性，事务一旦提交，它对数据库的改变就是**永久性的**，即使系统发生故障（断电、崩溃），数据也不会丢失。

> 在分布式事务下，更强调原子性。
### 分布式架构涉及事务的相关问题

在单体应用中，一个数据库事务（ACID）就能搞定。但在微服务/分布式架构下：

- 订单服务扣减库存
- 账户服务扣款
- 物流服务创建运单

这三个操作可能在三个不同的数据库里。你不能让"扣了款但没减库存"这种情况发生，所以需要分布式事务来保证**跨服务的一致性**。

## 分布式解决方案

### 分布式场景

在微服务/分布式架构下，典型的跨服务业务场景：

```
订单服务 ──扣减库存──▶ 库存服务 (DB-A)
   │
   ├──扣款──────▶ 账户服务 (DB-B)
   │
   └──创建运单──▶ 物流服务 (DB-C)
```

这三个操作分布在三个独立的数据库上。**任何一个环节失败，之前成功的操作都必须回滚**，否则就会出现"扣了款但没减库存"或"减了库存但没扣款"的数据不一致。

---

## 分布式事务解决方案全景

根据 CAP 理论，分布式事务的解决方案分两大阵营：

| 阵营        | 一致性级别         | 代表方案                       |
| --------- | ------------- | -------------------------- |
| **强一致性**  | 牺牲可用性，保证强一致   | XA/2PC、Seata AT            |
| **最终一致性** | 保证可用性，允许短暂不一致 | TCC、Saga、本地消息表、事务消息、最大努力通知 |

> 为了保证并发与性能， 通常采用非阻塞式的分布式事务，保证最终一致性。

---

## 一、强一致性方案

### 1. XA / 2PC（两阶段提交）

**原理**：基于数据库层协议，由事务协调者（TC）协调多个资源管理器（RM）。

```
         Coordinator
        /     |     \
     RM1     RM2     RM3
```

| 阶段                            | 动作                                                             |
| ----------------------------- | -------------------------------------------------------------- |
| **Phase 1 - Prepare**         | TC 向所有 RM 发送"准备提交"请求，RM 执行操作但不提交，锁定资源，返回 ready/abort           |
| **Phase 2 - Commit/Rollback** | 所有 RM 返回 ready → TC 发 commit 令所有 RM 提交；任一返回 abort → 发 rollback |

**Seata XA 模式**：Seata 对原生 XA 做了封装，与 AT/TCC/Saga 共用同一套 TC，降低了使用门槛。

| 优点 | 缺点 |
|------|------|
| ✅ 数据强一致，业务无感知 | ❌ **性能极差**：资源锁持有时间长，并发低 |
| ✅ 数据库原生支持，标准协议 | ❌ 单点故障：协调者宕机，所有参与者阻塞 |
| ✅ 适合单体拆微服务的过渡阶段 | ❌ 不支持跨数据库类型（MySQL → Redis） |

> **企业实践**：基本**不推荐**在互联网高并发场景使用。仅在银行核心账务等对一致性要求极致的场景仍有应用。

## 二、最终一致性方案

### 2. Seata AT 模式（Automatic Transaction）

**原理**：Seata 最核心的模式，通过数据源代理自动生成回滚日志（Undo Log），实现业务零侵入的分布式事务。

```
Phase 1：执行业务 SQL + 记录 Undo Log（同一本地事务），释放本地锁，注册全局锁
Phase 2（Commit）：异步删除 Undo Log
Phase 2（Rollback）：根据 Undo Log 反向补偿
```

| 优点                                     | 缺点                                |
| -------------------------------------- | --------------------------------- |
| ✅ **零侵入**：仅需 `@GlobalTransactional` 注解 | ❌ 仅支持关系型数据库（JDBC）                 |
| ✅ 自动生成回滚 SQL，无需手写补偿代码                  | ❌ 默认读未提交隔离级别（需 SELECT FOR UPDATE） |
| ✅ 性能远好于 XA                             | ❌ 全局锁有一定性能开销                      |
| ✅ 社区活跃，阿里巴巴背书                          | ❌ 不适合长事务（锁竞争加剧）                   |

**适用场景**：Java + 关系型数据库的标准微服务场景。Seata AT 是**目前国内使用最广泛的分布式事务方案**。

---

### 3. TCC 模式（Try-Confirm-Cancel）

**原理**：**业务层**的两阶段提交。每个服务需要实现三个接口：

```
Try：    预留资源（如冻结库存、预扣金额）
Confirm：确认提交（真正扣减冻结的库存）
Cancel： 取消释放（解冻库存、退还预扣金额）
```

**示例**（转账场景）：
```
账户A（转出）             账户B（转入）
Try:    冻结100元        Try:    预增100元（不可用）
Confirm: 扣除冻结的100元   Confirm: 确认入账100元
Cancel:  解冻100元        Cancel:  取消预增
```

| 优点                 | 缺点                           |
| ------------------ | ---------------------------- |
| ✅ 不依赖数据库事务，适用范围广   | ❌ **高侵入性**：每个服务写三个方法         |
| ✅ 锁粒度由业务控制，灵活      | ❌ 设计复杂，需处理空回滚、悬挂、幂等          |
| ✅ 锁时间短（仅 Try 阶段预留） | ❌ Confirm/Cancel 需幂等（可能重复调用） |
| ✅ 性能高，适合核心系统       | ❌ 开发成本高                      |

**TCC 三大异常处理**（必须自己解决）：
1. **空回滚**：Cancel 被调用时 Try 还未执行 —— 需判断资源是否已冻结
2. **悬挂**：Cancel 先于 Try 到达 —— Try 执行时需检查 Cancel 是否已执行
3. **幂等**：Confirm/Cancel 被重复调用 —— 需记录状态防重

**适用场景**：金融支付、交易核心链路等**对性能要求极高**的场景。TCC 是 Seata 性能天花板最高的模式。

---

### 4. Saga 模式

**原理**：将长事务拆分为多个**依次执行的本地事务**。每个事务完成后触发下一个。失败时通过**补偿事务**反向回滚。

```
订单 saga:
  T1: 创建订单 ──▶ T2: 扣减库存 ──▶ T3: 扣款 ──▶ T4: 创建运单
                    │
                    ▼（T3 失败）
  C1: 取消订单 ◀── C2: 恢复库存 ◀── ...  
```

**Seata Saga 状态机引擎**：用 JSON 定义状态流转，支持选择、并发、子状态机、重试、异常捕获等编排能力。

```json
{
  "Name": "orderSaga",
  "StartState": "CreateOrder",
  "States": {
    "CreateOrder": {
      "Type": "ServiceTask",
      "ServiceName": "orderService",
      "ServiceMethod": "create",
      "CompensateState": "CancelOrder",
      "Next": "ReduceStock"
    },
    "ReduceStock": { ... }
  }
}
```

| 优点                    | 缺点                     |
| --------------------- | ---------------------- |
| ✅ **高性能**：一阶段提交，无锁等待  | ❌ **无隔离性**：脏写可能使补偿无法回滚 |
| ✅ 事务可长时间运行（分钟~小时）     | ❌ 侵入性高，需定义 JSON + 补偿逻辑 |
| ✅ 高吞吐：参与者可异步执行        | ❌ 补偿逻辑设计复杂             |
| ✅ **跨系统友好**：仅发起方需要状态机 | ❌ 需处理空补偿、悬挂、幂等         |

**设计原则**：优先扣款后入账（宁可多扣不少给）。

**适用场景**：**长流程业务**（如订单流转、审批流程）、**跨公司/老旧系统集成**（合作伙伴无法实现 TCC 三接口）。

---

### 5. 本地消息表（Local Message Table）

**原理**：在业务数据库内建一张 `outbox` 消息表，**业务操作 + 写消息表**在同一个本地事务中完成。后台任务轮询消息表发送到 MQ。

```
┌──────────┐     本地事务       ┌──────────┐
│ 订单表   │◀────写入──────▶│ 消息表   │
└──────────┘                   └────┬─────┘
                                    │ 轮询发送
                                    ▼
                               ┌──────────┐
                               │  MQ      │
                               └────┬─────┘
                                    ▼
                               ┌──────────┐
                               │ 下游服务  │
                               └──────────┘
```

| 优点 | 缺点 |
|------|------|
| ✅ 实现简单，无外部依赖 | ❌ 耦合业务数据库（消息表占用存储） |
| ✅ 可靠投递：消息持久化到 DB | ❌ 轮询有延迟，不是实时的 |
| ✅ 天然幂等：消息 ID 可去重 | ❌ 高并发下轮询成为瓶颈 |

**适用场景**：对实时性要求不高、想快速实现最终一致性的中小规模系统。

---

### 6. 事务消息（Transactional Message / RocketMQ）

**原理**：RocketMQ 的**半消息（Half Message）**两阶段提交。消息先写入 MQ 但不可见，业务执行成功后提交消息使其可见。

```
阶段1 ── 发送半消息 ──▶ MQ 标记"待确认" ──▶ 对消费者不可见
阶段2 ── 执行本地事务 ──▶ 成功 → Commit → 消费者可见
                         失败 → Rollback → MQ 丢弃消息
阶段3 ── 事务状态回查（Recovery）：Broker 未收到确认时主动查询
```

**MQ 内部状态流转**：
```
Send Half Msg → 待提交 → [Commit] → 待消费 → [ACK] → 标记消费
                       → [Rollback] → 丢弃
                       → [超时] → 回查接口
```

**RocketMQ 回查机制**：
```java
// 生产者注册回查接口
producer.setTransactionListener(new TransactionListener() {
    // 执行本地事务
    LocalTransactionState executeLocalTransaction(Message msg, Object arg);
    // Broker 回查本地事务状态
    LocalTransactionState checkLocalTransaction(MessageExt msg);
});
```

| 优点 | 缺点 |
|------|------|
| ✅ **解耦**：不耦合业务数据库 | ❌ 强依赖 RocketMQ |
| ✅ 实时性好于本地消息表 | ❌ 需实现回查接口 |
| ✅ 高性能异步投递 | ❌ 一次事务仅支持一条消息 |
| ✅ 阿里云/社区成熟方案 | ❌ 下游消费者仍需幂等处理 |

**适用场景**：基于 RocketMQ 的微服务架构，需异步解耦且保证最终一致的场景。**阿里生态的标准方案**。

---

### 7. 最大努力通知（Best-Effort Notification）

**原理**：主动方向被动方发送通知，若失败则按**递增间隔重试**，直到达到最大重试次数。到达上限后介入人工处理。




```
通知方 ──(1)发送──▶ 接收方 ──❌ 失败
通知方 ◀──(2)重试 1min 后──▶ 接收方 ──❌ 失败  
通知方 ◀──(3)重试 5min 后──▶ 接收方 ──✅ 成功
...
通知方 ◀──(N)重试 1h 后───▶ 接收方 ──❌ 失败 → 告警/人工介入
```

**与事务消息的区别**：
| | 事务消息 | 最大努力通知 |
|---|---------|------------|
| **一致性要求** | 较高 | 较低 |
| **重试机制** | MQ 自带消息重投 | 业务方自行实现递增重试 |
| **回查/确认** | 有回查接口 | 被动方主动提供查询接口 |
| **典型场景** | 核心交易链路 | 支付结果通知、短信通知 |

**适用场景**：对一致性要求最低的场景，如**支付结果回调通知**、运营类通知。

## 三、方案全景对比

| 方案 | 一致性 | 吞吐量 | 侵入性 | 锁/资源占用 | 适用事务时长 | 社区成熟度 |
|------|--------|--------|--------|------------|-------------|-----------|
| **XA / 2PC** | 强一致 | ⭐ 低 | 低 | 高（全程锁） | 秒级 | 标准协议 |
| **Seata AT** | 弱一致/读未提交 | ⭐⭐⭐ 中 | **极低** | 中（全局锁） | 秒~分钟 | ⭐⭐⭐⭐⭐ |
| **TCC** | 最终一致 | ⭐⭐⭐⭐ 高 | **高** | 低（Try 预留） | 秒~分钟 | ⭐⭐⭐⭐ |
| **Saga** | 最终一致 | ⭐⭐⭐⭐⭐ 极高 | **高** | **无锁** | **分钟~小时** | ⭐⭐⭐⭐ |
| **本地消息表** | 最终一致 | ⭐⭐⭐ 中 | 中 | 无锁 | 秒~分钟 | ⭐⭐⭐ |
| **RocketMQ 事务消息** | 最终一致 | ⭐⭐⭐⭐ 高 | 中 | 无锁 | 秒~分钟 | ⭐⭐⭐⭐⭐ |
| **最大努力通知** | 弱一致 | ⭐⭐⭐⭐ 高 | 低 | 无锁 | 不限 | ⭐⭐⭐ |

## 四、企业选型指南

### 选型维度

```
选型五维模型：

    一致性要求
       ▲
       │
       │    XA/2PC
       │       │
       │    AT模式───── 侵入性
       │    ╱  ╲        ▲
       │  TCC  Saga      │
       │  ╱      ╲       │
       │事务消息  本地消息表
       ▼
    性能/吞吐
```

### 选型决策树

```
1. 是否需要强一致性？
   ├─ Yes → 能否接受低性能？
   │         ├─ Yes → XA/2PC（银行核心账务）
   │         └─ No  → Seata AT（有全局锁，读写未提交）
   └─ No  → 2. 事务时长多长？
              ├─ 秒~分钟 → 3. 是否用 Java + 关系型数据库？
              │              ├─ Yes → 4. 能否接受代码侵入？
              │              │         ├─ No/尽量少 → Seata AT ⭐推荐
              │              │         └─ Yes → TCC（高性能核心链路）
              │              └─ No/混合 → 5. 是否用 RocketMQ？
              │                            ├─ Yes → RocketMQ 事务消息
              │                            └─ No  → 本地消息表 / TCC
              └─ 分钟~小时 → 6. 是否可控所有参与方代码？
                              ├─ Yes → TCC 或 Saga
                              └─ No  → Saga（跨公司/老旧系统）
```

### 阿里系企业常见组合

| 业务场景 | 推荐方案 | 原因 |
|----------|---------|------|
| **常规交易**（下单、扣库存） | Seata AT | 零侵入、开发效率高 |
| **支付核心链路** | TCC | 高性能、业务自控资源 |
| **长流程**（订单生命周期、审批流） | Seata Saga | 无锁、长时间运行、跨系统 |
| **异步解耦**（积分、优惠券） | RocketMQ 事务消息 | 阿里生态标配 |
| **通知回调**（支付结果通知） | 最大努力通知 | 简单、可重试 |
| **金融对账**（核心账务） | XA/2PC | 强一致性要求 |

### 选择 Seata 的理由（来自社区数据）

根据 Apache Seata 官方文档和社区实践：

1. **AT 模式**：国内企业使用最多的分布式事务方案。适合 ≥80% 的微服务场景。
2. **TCC 模式**：金融/支付行业首选。阿里内部双十一核心链路使用。
3. **Saga 模式**：长事务编排。保险公司、航空票务等场景已被验证。
4. **XA 模式**：作为补充，兼容遗留系统。

## 五、总结

```
分布式事务选型核心原则：

1️⃣ 能不用分布式事务就不用
   → 业务聚合（DDD 聚合根）减少跨服务操作
   → 非核心链路允许不一致（对账修复）

2️⃣ 能用最终一致性就不用强一致性
   → 互联网场景下 CAP 优先保证 AP
   → 用户体感延迟 >> 数据瞬时不一致

3️⃣ 能选 AT 就不选 TCC
   → 零侵入 >> 手写补偿逻辑
   → 开发效率 >> 极致性能（大多数场景）

4️⃣ 长事务用 Saga，异步解耦用事务消息
   → 锁是性能杀手，长事务必须无锁
   → MQ 解耦是微服务的核心范式
```

**推荐学习路径**：Seata AT → RocketMQ 事务消息 → TCC → Saga → XA
